Título do Documento: Especificação de Validação de Campo CardDemo

Descrição Resumida: Este documento descreve a funcionalidade de um trecho de código dentro do aplicativo CardDemo, com foco em como o sistema fornece feedback visual aos usuários durante a entrada de dados. O código garante a precisão dos dados, destacando possíveis erros e orientando os usuários para o preenchimento de campos obrigatórios.

Histórias do Usuário: Como representante de atendimento ao cliente, preciso que o sistema indique claramente quando cometi um erro ou perdi um campo obrigatório, para que eu possa inserir dados com precisão e evitar que informações incorretas sejam armazenadas no sistema.

Épico Relacionado: 9 - Utilitários do Sistema

Requisitos Técnicos:

  • Validação de Campo: Este bloco de código valida a entrada do usuário no aplicativo CardDemo, verificando especificamente as condições de erro e campos em branco.
    • Entrada: O código recebe a entrada de uma variável (TESTVAR1). Esta variável parece conter um sinalizador ou código de status indicando o resultado de uma validação anterior ou operação de recuperação de dados. Ele também depende de outra variável, provavelmente um sinalizador, chamada CDEMO-PGM-REENTER, para determinar se o usuário está inserindo dados novamente após uma tentativa anterior de correção de erro.
    • Destaque Condicional: O cerne da lógica está no bloco condicional.
      • Ele primeiro verifica se (TESTVAR1) indica um erro (FLG-(TESTVAR1)-NOT-OK) ou se o campo está em branco (FLG-(TESTVAR1)-BLANK).
      • Além disso, ele verifica se o usuário está inserindo dados novamente (CDEMO-PGM-REENTER) para evitar destacar campos desnecessariamente antes que o usuário tenha a chance de corrigi-los.
    • Feedback Visual: Se ambas as condições forem atendidas, ele aplica feedback visual ao campo na tela representado por (SCRNVAR2). Ele usa DFHRED, provavelmente uma constante do sistema que representa a cor vermelha, para destacar o campo, sinalizando um erro ao usuário.
    • Tratamento de Campo em Branco: Se o campo estiver em branco (FLG-(TESTVAR1)-BLANK), ele também marcará o campo com um asterisco (*) dentro do próprio campo ((SCRNVAR2)O OF (MAPNAME3)O).
    • Saída: O código não produz saída direta, mas modifica a exibição da tela alterando a cor e o conteúdo do campo (SCRNVAR2) com base nos resultados da validação.

Configurações:

  • DFHRED: Constante ou variável do sistema que representa a cor vermelha.
  • (MAPNAME3): Mapa de tela ou definição de layout dentro do aplicativo CardDemo, especificando a organização e aparência dos campos na tela.

Melhorias de Código:

  • Registro de Erros: Implementar o registro de erros de validação em um arquivo ou banco de dados. Este log pode ser inestimável para depuração e rastreamento de problemas recorrentes.
  • Mensagens de Erro: Em vez de apenas destacar o campo, exiba uma mensagem de erro mais amigável perto do campo, fornecendo detalhes específicos sobre o erro.
  • Configuração: Considere mover as regras de validação para um arquivo de configuração. Isso permitiria uma modificação mais fácil da lógica de validação sem alterações no código.

Melhorias de Segurança:

  • Limpeza de Entrada: Sempre limpe a entrada do usuário antes da validação para evitar ataques de injeção. Isso é especialmente importante se os dados de entrada forem usados posteriormente em consultas de banco de dados ou exibidos para outros usuários.
  • Registro Seguro: Se o registro de erros for implementado, certifique-se de que dados confidenciais, como entrada do usuário, sejam mascarados ou hash para evitar exposição.
